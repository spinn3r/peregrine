28  /**
29   * 
30   * http://www.xaprb.com/blog/2010/01/09/how-linux-iostat-computes-its-results/
31   * 
32   * iostat is one of the most important tools for measuring disk performance, which
33   * of course is very relevant for database administrators, whether your chosen
34   * database is Postgres, MySQL, Oracle, or anything else that runs on
35   * GNU/Linux. Have you ever wondered where statistics like await (average wait for
36   * the request to complete) come from? If you look at the disk statistics the Linux
37   * kernel makes available through files such as /proc/diskstats, you won't see
38   * await there. How does iostat compute await? For that matter, how does it compute
39   * the average queue size, service time, and utilization? This blog post will show
40   * you how that's computed.
41   * 
42   * First, let's look at the fields in /proc/diskstats. The order and location
43   * varies between kernels, but the following applies to 2.6 kernels. For reads and
44   * writes, the file contains the number of operations, number of operations merged
45   * because they were adjacent, number of sectors, and number of milliseconds
46   * spent. Those are available separately for reads and writes, although iostat
47   * groups them together in some cases. Additionally, you can find the number of
48   * operations in progress, total number of milliseconds during which I/Os were in
49   * progress, and the weighted number of milliseconds spent doing I/Os. Those are
50   * not available separately for reads and writes.
51   * 
52   * The last one is very important. The field showing the number of operations in
53   * progress is transient - it shows you the instantaneous value, and this
54   * "memoryless" property means you can't use it to infer the number of I/O
55   * operations that are in progress on average. But the last field has memory,
56   * because it is defined as follows:
57   * 
58   * Field 11 - weighted # of milliseconds spent doing I/Os This field is incremented
59   * at each I/O start, I/O completion, I/O merge, or read of these stats by the
60   * number of I/Os in progress (field 9) times the number of milliseconds spent
61   * doing I/O since the last update of this field. This can provide an easy measure
62   * of both I/O completion time and the backlog that may be accumulating.
63   * 
64   * So the field indicates the total number of milliseconds that all requests have
65   * been in progress. If two requests have been waiting 100ms, then 200ms is added
66   * to the field. And thus it records what happened over the duration of the
67   * sampling interval, not just what's happening at the instant you look at the
68   * file. We'll come back to that later.
69   * 
70   * Now, given two samples of I/O statistics and the time elapsed between them, we
71   * can easily compute everything iostat outputs in -dx mode. I'll take them
72   * slightly out of order to reflect how the computations are done internally.
73   * 
74   * rrqm/s is merely the incremental merges divided by the number of seconds elapsed.
75   * 
76   * wrqm/s is similarly simple, and r/s, w/s, rsec/s, and wsec/s are too.
77   * 
78   * avgrq-sz is the number of sectors divided by the number of I/O operations.
79   * 
80   * avgqu-sz is computed from the last field in the file - the one that has "memory"
81   * - divided by the milliseconds elapsed. Hence the units cancel out and you just
82   * get the average number of operations in progress during the time period. The
83   * name (short for "average queue size") is a little bit ambiguous. This value
84   * doesn't show how many operations were queued but not yet being serviced - it
85   * shows how many were either in the queue waiting, or being serviced. The exact
86   * wording of the kernel documentation is "as requests are given to appropriate
87   * struct request_queue and decremented as they finish."
88   * 
89   * %util is the total time spent doing I/Os, divided by the sampling interval. This
90   * tells you how much of the time the device was busy, but it doesn't really tell
91   * you whether it's reaching its limit of throughput, because the device could be
92   * backed by many disks and hence capable of multiple I/O operations
93   * simultaneously.
94   * 
95   * await is the total time for all I/O operations summed, divided by the number of
96   * I/O operations completed.
97   * 
98   * svctm is the most complex to derive. It is the utilization divided by the
99   * throughput. You saw utilization above; the throughput is the number of I/O
100  * operations in the time interval.
101  * 
102  * Although the computations and their results seem both simple and cryptic, it
103  * turns out that you can derive a lot of information from the relationship between
104  * these various numbers. This is one of those tools where a few lines of code have
105  * a surprising amount of meaning, which is left for the reader to understand. I'll
106  * get more into that in the future.
107  * 
108  */

113     public LinuxSystemProfiler( Set<String> interfaces,
114                                 Set<String> disks,
115                                 Set<String> processors )
116         throws IOException {
117 
118         setInterfaces( interfaces );
119         setProcessors( processors );
120         
121         // see if the disks need to be looked at ...
122         Set<String> tmp = new HashSet();
123 
124         for ( String disk : disks ) {
125 
126             disk = disk.replaceAll( "/dev/" , "" );
127 
128             if ( disk.startsWith( "/" ) ) {
129                 // this is a mount point so resolve it to a specific path.
130                 disk = resolveMountPoint( disk );
131 
132                 if( disk != null ) {
133                     log.debug( "Mount point resolved to: %s" , disk );
134                 } else {
135                     log.warn( "Mount point NOT resolved" );
136                 }
137                 
138             }
139                 
140             tmp.add( disk );
141 
142         }
143 
144         setDisks( tmp );
145 
146     }

178     /**
179      * Fields are documented here:
180      * 
181      * http://www.mjmwired.net/kernel/Documentation/iostats.txt
182      *
183      * An example line is here.
184      *
185      * 8 0 sda 1663183 50370 286417462 8075722 16114577 124388899 1123914366 612454228 0 23049653 620527121
186      *
187      * The primary ones we care about are:
188      * 
189      * 
190      * Field  3 -- # of sectors read
191      * Field  7 -- # of sectors written
192      *
193      * For reference ALL the fields are here:
194      * 
195      *	Field  1 -- # of reads completed
196      *	    This is the total number of reads completed successfully.
197      *	Field  2 -- # of reads merged, field 6 -- # of writes merged
198      *	    Reads and writes which are adjacent to each other may be merged for
199      *	    efficiency.  Thus two 4K reads may become one 8K read before it is
200      *	    ultimately handed to the disk, and so it will be counted (and queued)
201      *	    as only one I/O.  This field lets you know how often this was done.
202      *	Field  3 -- # of sectors read
203      *	    This is the total number of sectors read successfully.
204      *	Field  4 -- # of milliseconds spent reading
205      *	    This is the total number of milliseconds spent by all reads (as
206      *	    measured from __make_request() to end_that_request_last()).
207      *	Field  5 -- # of writes completed
208      *	    This is the total number of writes completed successfully.
209      *	Field  7 -- # of sectors written
210      *	    This is the total number of sectors written successfully.
211      *	Field  8 -- # of milliseconds spent writing
212      *	    This is the total number of milliseconds spent by all writes (as
213      *	    measured from __make_request() to end_that_request_last()).
214      *	Field  9 -- # of I/Os currently in progress
215      *	    The only field that should go to zero. Incremented as requests are
216      *	    given to appropriate struct request_queue and decremented as they finish.
217      *	Field 10 -- # of milliseconds spent doing I/Os
218      *	    This field increases so long as field 9 is nonzero.
219      *	Field 11 -- weighted # of milliseconds spent doing I/Os
220      *	    This field is incremented at each I/O start, I/O completion, I/O
221      *	    merge, or read of these stats by the number of I/Os in progress
222      *	    (field 9) times the number of milliseconds spent doing I/O since the
223      *	    last update of this field.  This can provide an easy measure of both
224      *	    I/O completion time and the backlog that may be accumulating.
225      * 
226      */
227     private void captureDisk( StatMeta statMeta ) throws IOException {
228 
229         String value = read( "/proc/diskstats" );
230         
231         String[] lines = value.split( "\n" );
232 
233         for( String line : lines ) {
234             
235             String[] fields = line.split( "[\t ]+" );
236 
237             if ( fields.length < 10 ) {
238                 continue;
239             }
240 
241             // the name of the device.
242             String field_disk                 = fields[3];
243             String field_reads                = fields[4];
244             String field_readsMerged          = fields[5];
245             String field_sectorsRead          = fields[6];
246             String field_timeSpentReading     = fields[7];
247             String field_writes               = fields[8];
248             String field_writesMerged         = fields[9];   
249             String field_sectorsWritten       = fields[10];  // Field 7
250             String field_timeSpentWriting     = fields[11];  // Field 8
251             String field_pending              = fields[12];  // Field 9
252             String field_time                 = fields[13];  // Field 10
253 
254             if ( getDisks() != null && ! getDisks().contains( field_disk ) ) {
255                 continue;
256             }
257 
258             DiskStat stat = new DiskStat();
259             stat.timestamp = statMeta.timestamp;
260             stat.name = field_disk;
261 
262             stat.reads             = new BigDecimal( field_reads );
263             stat.writes            = new BigDecimal( field_writes );
264 
265             stat.readBytes         = sectorReference( field_sectorsRead );
266             stat.writtenBytes      = sectorReference( field_sectorsWritten );
267             stat.timeSpentReading  = new BigDecimal( field_timeSpentReading );
268             stat.timeSpentWriting  = new BigDecimal( field_timeSpentWriting );
269             stat.time              = new BigDecimal( field_time );
270 
271             stat.readsMerged       = new BigDecimal( field_readsMerged );
272             stat.writesMerged      = new BigDecimal( field_writesMerged );
273 
274             stat.init();
275             
276             statMeta.diskStats.add( stat );
277             
278         }
279 
280     }

282     private void captureInterface( StatMeta statMeta ) throws IOException {
283 
284         String value = read( "/proc/net/dev" );
285 
286         String[] lines = value.split( "\n" );
287 
288         for( int i = 0; i < lines.length; ++i ) {
289 
290             if ( i <= 1 )
291                 continue;
292 
293             String line = lines[i];
294 
295             String[] fields = line.split( "[\t ]+" );
296 
297             String field_net = fields[1];
298             field_net = field_net.substring( 0 , field_net.length() - 1 );
299 
300             if ( fields.length < 11 )
301                 continue;
302             
303             String field_receive_bytes  = fields[2];
304             String field_transmit_bytes = fields[10];
305 
306             if ( getInterfaces() != null && ! getInterfaces().contains( field_net ) ) {
307                 continue;
308             }
309 
310             InterfaceStat stat = new InterfaceStat();
311             stat.timestamp = statMeta.timestamp;
312             stat.name = field_net;
313 
314             stat.readBits = new BigDecimal( field_receive_bytes )
315                 .multiply( new BigDecimal( 8 ) )
316                 ;
317             
318             stat.writtenBits = new BigDecimal( field_transmit_bytes )
319                 .multiply( new BigDecimal( 8 ) )
320                 ;
321 
322             stat.init();
323             
324             statMeta.interfaceStats.add( stat );
325             
326         }

330     /**
331      * http://www.linuxhowtos.org/System/procstat.htm
332      * http://stackoverflow.com/questions/3017162/how-to-get-total-cpu-usage-in-linux-c
333      */
334     private void captureProcessor( StatMeta statMeta ) throws IOException {
335 
336         String value = read( "/proc/stat" );
337 
338         String[] lines = value.split( "\n" );
339 
340         for( String line : lines ) {
341             
342             String[] fields = line.split( "[\t ]+" );
343 
344             String field_cpu     = fields[0];
345 
346             if ( ! field_cpu.startsWith( "cpu" ) )
347                 continue;
348 
349             // user: normal processes executing in user mode
350             // nice: niced processes executing in user mode
351             // system: processes executing in kernel mode
352             // idle: twiddling thumbs
353             // iowait: waiting for I/O to complete
354             // irq: servicing interrupts
355             // softirq: servicing softirqs
356 
357             ProcessorStat stat = new ProcessorStat();
358             stat.timestamp = statMeta.timestamp;
359 
360             stat.name     = field_cpu;
361             stat.user     = new BigDecimal( fields[1] );
362             stat.nice     = new BigDecimal( fields[2] );
363             stat.system   = new BigDecimal( fields[3] );
364             stat.idle     = new BigDecimal( fields[4] );
365             stat.iowait   = new BigDecimal( fields[5] ); 
366             stat.irq      = new BigDecimal( fields[6] );
367             stat.softirq  = new BigDecimal( fields[7] );
368 
369             stat.init();
370 
371             statMeta.processorStats.add( stat );
372             
373         }
374             
375     }

383     public String resolveMountPoint( String path ) throws IOException {
384 
385         // resolve symlinks, etc.
386         path = new File( path ).getCanonicalPath();
387         
388         String value = read( "/proc/mounts" );
389         
390         String[] lines = value.split( "\n" );
391 
392         String result_device = null;
393         String result_mntpoint = "";
394 
395         for( String line : lines ) {
396 
397             String[] fields = line.split( "[\t ]+" );
398 
399             String device = fields[0];
400             String mntpoint = fields[1];
401             
402             if ( path.indexOf( mntpoint ) == 0 ) {
403 
404                 if ( mntpoint.length() > result_mntpoint.length() ) {
405                     result_device = device.replaceAll( "/dev/", "" );
406                     result_mntpoint = mntpoint;
407                 }
408 
409             }
410 
411         }
412 
413         return result_device;
414         
415     }

417     /**
418      * Perform a consistent read on a /proc file.
419      */
420     private String read( String path ) throws IOException {
421 
422         //TODO: move this to a dedicated function in Files or somewhere outside
423         //this package.
424 
425         File file = new File( path );
426         
427         RandomAccessFile fis = null;
428 
429         try {
430 
431             fis = new RandomAccessFile( file, "r" );
432 
433             byte[] buff = new byte[16384];
434 
435             int count = fis.read( buff );
436 
437             byte[] result = new byte[ count ];
438             System.arraycopy( buff, 0, result, 0, count );
439 
440             return new String( result );
441 
442         } finally {
443             new Closer( fis ).close();
444         }
445 
446     }
